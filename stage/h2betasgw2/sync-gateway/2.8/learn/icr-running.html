Running Replications

  • Under Development
  • beta -- H2-SGW as at 10Jul20
    +
    Author’s Notes
    CONTENT UNDER DEVELOPMENT

    This content covers how to run inter-cluster replication (replication between Sync Gateway clusters)

    Related — Configuration Schema | Admin REST API

    Topic Group — Inter-cluster Replication

    Running via the REST API

    A replication is run by sending a POST request to the server endpoint replication, with a JSON object defining the replication parameters. Both one-shot and continuous replications can be run.

    Each replication is one-way between two local or a local and a remote Sync Gateway databases.

    Multiple replications can run simultaneously, supporting bi-directional replications and different replication topologies. Be aware that both databases being synchronized should have the same sync function, otherwise it could lead to unexpected behavior.

    You can submit the request using the curl utility or an application such as Postman

    Example 1. Submitting API Requests

    • Using CURL

    • Using Postman

    curl -X PUT "http://localhost:4985/db1/_replication/rep1" -H "accept: application/json" -H "Content-Type: application/json" -d "{ \"adhoc\": false, \"batch_size\": 0, \"conflict_resolution_type\": \"string\", \"continuous\": false, \"custom_conflict_resolver\": \"string\", \"direction\": \"string\", \"enable_delta_sync\": false, \"filter\": \"string\", \"max_backoff_time\": 0, \"perf_tuning_params\": [ \"string\" ], \"purge_on_removal\": false, \"query_params\": [ \"string\" ], \"remote\": \"string\", \"replication_id\": \"string\", \"state\": \"string\"}"

    • One-shot

    • Continuous

    These parameters start a one-shot replication between one database on the local Sync Gateway instance and one on a remote Sync Gateway instance, with user credentials. The request will return immediately and the replication will run asynchronously.

    {
  "replication_id": "our-rep-1",
  "direction": "pull",
  "remote": "http://user:password@example.com:4985/db1",
  "filter":"sync_gateway/bychannel",
  "query_params": {
    "channels": ["channel1.user1"]
  }
}

    These parameters start a continuous replication between one database on the local Sync Gateway instance and one on a remote Sync Gateway instance. The replicator will batch up to 1000 revisions at a time, this will improve replication performance but will use more memory resources. Source database documents will be filtered so that only those tagged with the channel names "channel1" or "channel2" are replicated.

    {
  "replication_id": "our-rep1",
  "direction": "pull",
  "purge-on-removal": true,
  "remote": "http://user:password@example.com:4985/db1",
  "filter":"sync_gateway/bychannel",
  "query_params": {
    "channels": ["channel1.user1"]
  },
  "continuous": true

}

    See: Configuration Schema for a full description of the available configuration properties.

    You will find the replication properties defined here: replications.

    Running replication on startup

    If you want to run replications as soon as Sync Gateway starts, you can define replications in the JSON configuration file — see: Configuration Schema for a full description of the available configuration properties.

    Replications are defined in the context of a local database; that is they are defined under the databases.<db> property. You will find the replication properties defined here: replications.

    The replications value is an array of objects. Each object defines a single replication. The object properties are, in the main, the same as those for the replication end-point on the Admin REST API.

    One-shot replications are always run asynchronously even if the async property is not set to true.

    One-shot replications are run after a short delay (5 seconds), which allows the local REST API’s to come up.

    Replications may be given a user defined replication_id otherwise Sync Gateway will generate a random UUID.

    Replications defined in config may not contain the cancel property.

    Example 2. Simple replication config
    //
{
//  ... may be preceded by additional configuration data as required by the user ...

"databases": {
  "db": {
    "replications": {
      "repl_id_1": {
        "remote": "https://remoteSG:4984/remoteDB",
      	"direction": "pull",
      	"purge_on_removal":true,
      	"continuous":true,
      	"filter":"sync_gateway/bychannel",
      	"conflict_resolution_type":"custom",
      	"custom_conflict_resolver": `func(local, remote) {...}`,
      }

    }
  }
  //  ... may be followed by additional configuration data as required by the user ...
}
//
