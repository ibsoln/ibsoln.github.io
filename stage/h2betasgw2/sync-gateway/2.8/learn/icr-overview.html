Inter-Cluster Replication

  • Under Development
  • beta -- H2-SGW as at 10Jul20

    Inter-cluster replication (Sync Gateway to Sync Gateway) using SG-Replicate 2.0 protocol

    This content provides an overview of Inter-cluster replication, which is replication between Sync Gateway clusters using SG-Replicate 2.0.

    Related — Configuration Schema | Admin REST API

    Topic Group — Inter-cluster Replication

    Overview

    Couchbase Sync Gateway supports inter-cluster replication, [1] an increasingly important enterprise-level requirement that supports cloud-to-edge sync synchronization use cases, where data changes must be synchronized between a centralized cloud cluster and a large number of edge clusters whilst still enforcing fine grained access control.

    Inter-cluster replication provides the ability to run replications between:

    • Sync Gateway clusters

    • Active mobile clusters.

    Sync Gateway’s inter-cluster replication feature uses the SG-Replicate protocol. It is an integral part of Sync Gateway, acting as the active replicator, pulling changes from a data source and pushing them to a target.

    Example 1. Inter-cluster replication architecture

    All documents pass through the target Sync Gateway’s sync function instance, ensuring that access rules are honored. See: Work With Sync Functions for more on using sync functions)

    In the architecture diagram (Example 1), any database changes made on either Sync Gateway instance are replicated to the other Sync Gateway instance, in accordance with the replication’s configuration — see replications for configuration detail.

    Replication Types

    Sync Gateway supports two replication types

    Persistent

    Persistent replications survive Sync Gateway node restarts. This is the default replication-type.

    All replications created by configuring them in the Configuration Schema file are persistent. This is the recommended way to create replications in a production environment.

    You can also create persistent replications by using the Admin REST API.

    Ad hoc

    Ad hoc replications exist only for the period of the replication. They can be created using the Admin REST API. Once created they start in the 'running' state. When they enter 'stopped', or 'error', state they are automatically removed.

    Ad hoc replications provide a convenient way to:

    • Run one off replications (for example, when troubleshooting)

    • Run on-demand replications after Sync Gateway is started. For instance a replication that needs to be to be run only periodically can be configured as an ad hoc replication by an automated script scheduled to run when needed.

    Use Cases

    Cloud-to-edge synchronization

    In this multi-cloud deployment mode large numbers of multiple edge clusters sync with one or more clusters in cloud data centers. Each edge can operate autonomously without network connectivity to the cloud data centers [2].

    Conflict Resolution

    Sync Gateway’s inter-cluster replication supports automatic conflict resolution to resolve conflicting document changes (revisions). It delivers this using a flexible policy-driven mechanism, which automatically detects and resolves document conflicts under replication.

    A set of conflict resolver policies are available out-of-the-box. They can be easily applied to Sync Gateway replications to give automatic resolution.

    As soon as the active Sync Gateway database detects a conflict in a replicated document revision, it applies the specified resolver policy. This policy assesses the conflicting revisions and either determines the winning revision or returns an error if it fails while doing so.

    See: Conflict Resolution
